title: "Output format"
description: "Reference for COLMAP's sparse and dense reconstruction output files, including text and binary formats, camera conventions, and format conversion."
---

COLMAP stores reconstructions in a directory containing several files. By default it uses a binary format (fast, machine-readable). A human-readable text format is also supported. Both formats carry the same information split across files for rigs, cameras, frames, images, and 3D points.

<Note>
When a directory contains both binary and text versions of the model files, COLMAP prefers the binary format.
</Note>

## Sparse reconstruction

### Text format

COLMAP exports the following text files for every reconstructed model. Comments start with `#` and are ignored. The first comment lines briefly describe the format of each file.

<AccordionGroup>
  <Accordion title="rigs.txt">
    Contains the configured rigs and sensors. One line of data per rig.

    ```
    # Rig calib list with one line of data per calib:
    #   RIG_ID, NUM_SENSORS, REF_SENSOR_TYPE, REF_SENSOR_ID,
    #   SENSORS[] as (SENSOR_TYPE, SENSOR_ID, HAS_POSE,
    #                 [QW, QX, QY, QZ, TX, TY, TZ])
    # Number of rigs: 1
    1 2 CAMERA 1 CAMERA 2 1 -0.9999701516465348 -0.001112 -0.007534 0.001298 -0.19316 0.000852 0.007075
    2 1 CAMERA 3
    ```

    The first rig has two cameras; the second has one.
  </Accordion>

  <Accordion title="cameras.txt">
    Contains the intrinsic parameters of all reconstructed cameras. One line per camera.

    ```
    # Camera list with one line of data per camera:
    #   CAMERA_ID, MODEL, WIDTH, HEIGHT, PARAMS[]
    # Number of cameras: 3
    1 SIMPLE_PINHOLE 3072 2304 2559.81 1536 1152
    2 PINHOLE 3072 2304 2560.56 2560.56 1536 1152
    3 SIMPLE_RADIAL 3072 2304 2559.69 1536 1152 -0.0218531
    ```

    The length of `PARAMS[]` is variable and depends on the camera model. For `SIMPLE_PINHOLE`, there are 3 parameters: a single focal length and the principal point `(cx, cy)`. Camera intrinsics can be shared by multiple images, which reference cameras by `CAMERA_ID`.
  </Accordion>

  <Accordion title="frames.txt">
    Contains frames, where a frame is a specific instance of a rig with all or a subset of sensors exposed at the same time. One line per frame.

    ```
    # Frame list with one line of data per frame:
    #   FRAME_ID, RIG_ID, RIG_FROM_WORLD[QW, QX, QY, QZ, TX, TY, TZ],
    #   NUM_DATA_IDS, DATA_IDS[] as (SENSOR_TYPE, SENSOR_ID, DATA_ID)
    # Number of frames: 151
    1 1 0.998013 0.040985 0.041890 -0.023111 -5.26665 -0.17120 0.12300 2 CAMERA 1 1 CAMERA 2 2
    2 2 0.998164 0.037605 0.043101 -0.019881 -5.19560 -0.20794 0.14967 1 CAMERA 3 3
    ```
  </Accordion>

  <Accordion title="images.txt">
    Contains the pose and keypoints of all reconstructed images. **Two lines per image**: the first line contains the pose and metadata, the second line contains the 2D keypoint observations.

    ```
    # Image list with two lines of data per image:
    #   IMAGE_ID, QW, QX, QY, QZ, TX, TY, TZ, CAMERA_ID, NAME
    #   POINTS2D[] as (X, Y, POINT3D_ID)
    # Number of images: 2, mean observations per image: 2
    1 0.851773 0.0165051 0.503764 -0.142941 -0.737434 1.02973 3.74354 1 P1180141.JPG
    2362.39 248.498 58396 1784.7 268.254 59027 1784.7 268.254 -1
    2 0.851773 0.0165051 0.503764 -0.142941 -0.737434 1.02973 3.74354 1 P1180142.JPG
    1190.83 663.957 23056 1258.77 640.354 59070
    ```

    A `POINT3D_ID` of `-1` means the keypoint does not observe any reconstructed 3D point.
  </Accordion>

  <Accordion title="points3D.txt">
    Contains all reconstructed 3D points. One line per point.

    ```
    # 3D point list with one line of data per point:
    #   POINT3D_ID, X, Y, Z, R, G, B, ERROR, TRACK[] as (IMAGE_ID, POINT2D_IDX)
    # Number of points: 3, mean track length: 3.3334
    63390 1.67241 0.292931 0.609726 115 121 122 1.33927 16 6542 15 7345 6 6714 14 7227
    63376 2.01848 0.108877 -0.026084 102 209 250 1.73449 16 6519 15 7322 14 7212 8 3991
    63371 1.71102 0.28566 0.53475 245 251 249 0.612829 118 4140 117 4473
    ```

    - `POINT3D_ID` — unordered, non-contiguous identifier
    - `X, Y, Z` — 3D position
    - `R, G, B` — color (0–255)
    - `ERROR` — reprojection error in pixels (updated after global bundle adjustment)
    - `TRACK[]` — list of `(IMAGE_ID, POINT2D_IDX)` pairs; `POINT2D_IDX` is the zero-based index into that image's keypoint list
  </Accordion>
</AccordionGroup>

### Binary format

All binary files use **little-endian byte ordering**. The binary format is faster to read and write than the text format, making it the default. File extensions are `.bin` instead of `.txt`:

- `rigs.bin`
- `cameras.bin`
- `frames.bin`
- `images.bin`
- `points3D.bin`

The binary format can be parsed most conveniently using the C++ reconstruction API in `src/colmap/scene/reconstruction_io.h` or using pycolmap.

<Note>
Older COLMAP versions had no rig support, so `rigs` and `frames` files may be absent in legacy models. The I/O routines handle this automatically by initializing trivial rigs and frames.
</Note>

## Indices and identifiers

| Naming pattern | Meaning |
|---|---|
| `*_idx` | Ordered, contiguous zero-based index |
| `*_id` | Unordered, non-contiguous identifier |

For example, `CAMERA_ID`, `IMAGE_ID`, and `POINT3D_ID` are unordered identifiers. The maximum `POINT3D_ID` does not necessarily equal the number of 3D points, because some IDs are skipped during filtering.

## Camera extrinsics convention

The reconstructed pose of an image represents the transformation **from world to camera** (camera-from-world). It is stored as:

- A unit quaternion `(QW, QX, QY, QZ)` in Hamilton convention (used by Eigen)
- A translation vector `(TX, TY, TZ)`

The **camera center** (projection center) in world coordinates is:

```
camera_center = -R^T * T
```

where `R` is the 3×3 rotation matrix composed from the quaternion and `T` is the translation vector.

The local camera coordinate system is defined as:
- X axis — points right
- Y axis — points down
- Z axis — points forward (into the scene)

## Dense reconstruction output

The dense reconstruction workspace has the following structure:

```
+── images/                      # Undistorted images
+── sparse/
│   +── cameras.txt
│   +── images.txt
│   +── points3D.txt
+── stereo/
│   +── consistency_graphs/
│   │   +── image1.jpg.photometric.bin
│   +── depth_maps/
│   │   +── image1.jpg.photometric.bin
│   +── normal_maps/
│   │   +── image1.jpg.photometric.bin
│   +── patch-match.cfg
│   +── fusion.cfg
+── fused.ply                    # Fused point cloud
+── meshed-poisson.ply           # Poisson surface mesh
+── meshed-delaunay.ply          # Delaunay surface mesh
+── textured/
│   +── mesh.ply                 # Mesh with per-face UV coordinates
│   +── texture.png              # Texture atlas
+── run-colmap-geometric.sh
+── run-colmap-photometric.sh
```

### Depth and normal maps

Depth and normal maps are stored as mixed text/binary files. The text header specifies dimensions in the format `width&height&channels&` followed by row-major `float32` binary data.

- Depth maps: `channels=1`
- Normal maps: `channels=3`

These files can be read with pycolmap in Python or with the Matlab scripts at `scripts/matlab/read_depth_map.m` and `scripts/matlab/read_normal_map.m`.

### Consistency graphs

The consistency graph encodes, for each pixel in an image, which source images that pixel is consistent with. The binary part is a list of `int32` values in the format `<row><col><N><image_idx1>...<image_idxN>`. The image indices reference the ordering in `images.txt`.

## Converting between formats

To convert between binary and text format using the GUI:

<Steps>
  <Step title="Import the model">
    Choose **File > Import model** and select the folder containing the reconstruction files.
  </Step>
  <Step title="Export in the target format">
    - **File > Export model** — exports in binary format
    - **File > Export model as text** — exports in text format
    - **File > Export as...** — exports to other formats (NVM, Bundler, PLY, VRML)
  </Step>
</Steps>

To convert from the command line, use `model_converter`:

```bash
# Binary to text
colmap model_converter \
    --input_path /path/to/model \
    --output_path /path/to/output \
    --output_type TXT

# Text to binary
colmap model_converter \
    --input_path /path/to/model \
    --output_path /path/to/output \
    --output_type BIN

# Export to NVM (VisualSfM)
colmap model_converter \
    --input_path /path/to/model \
    --output_path /path/to/output \
    --output_type NVM

# Export to Bundler
colmap model_converter \
    --input_path /path/to/model \
    --output_path /path/to/output \
    --output_type BUNDLER

# Export to PLY point cloud
colmap model_converter \
    --input_path /path/to/model \
    --output_path /path/to/output \
    --output_type PLY

# Export to VRML
colmap model_converter \
    --input_path /path/to/model \
    --output_path /path/to/output \
    --output_type VRML
```

Supported output types: `BIN`, `TXT`, `NVM`, `BUNDLER`, `PLY`, `VRML`.

## Importing and exporting in the GUI

- To export the currently selected model: **File > Export model**
- To export all reconstructed models in the dataset: **File > Export all**
- To import an exported model (for visualization or to resume reconstruction): **File > Import model**, then select the folder containing the reconstruction files

The exported folder also contains the current project configuration for convenient re-import into COLMAP.
